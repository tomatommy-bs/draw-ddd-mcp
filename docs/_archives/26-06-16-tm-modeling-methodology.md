# TM（T字形ER）モデリング手法

## 概要

TM（T字形ER手法）は、佐藤正美氏が提唱したデータモデリング手法である。
リレーショナルモデルの数学的基盤（集合論・述語論理）に立脚しながら、
実務上の要件からデータ構造を**機械的・演繹的に**導出できる点に最大の特徴がある。

従来の ER モデリングが設計者の経験や直感に依存するのに対し、
TM は**分類規則と導出規則**を明示しており、同じ要件から同じモデルに到達しやすい。

---

## 基本原理

### 二大分類：Resource と Event

TM ではすべてのエンティティを **2種類** に分類する。これが手法全体の土台となる。

| 分類 | 定義 | 個体指定子の特徴 | 例 |
|------|------|------------------|-----|
| **Resource (R)** | 独立して存在するもの | 単独の識別子 | 顧客、商品、倉庫、社員 |
| **Event (E)** | Resource 間の対応関係を記録するもの | 複合識別子（R の識別子の組み合わせ） | 受注、出荷、請求、配属 |

この分類は恣意的なものではなく、**個体指定子（identifier）の構造** によって決定される。

### 個体指定子による分類規則

```
個体指定子が単独（1つのコード体系で個体を特定できる）
  → Resource

個体指定子が複合（2つ以上の Resource の識別子を含む）
  → Event
```

例：
- `顧客コード` だけで一意に特定 → **Resource**
- `顧客コード` + `商品コード` + `受注日` で一意に特定 → **Event**

### T字形の由来

Event が2つの Resource を結ぶ形が「T字」に見えることに由来する。

```
  Resource A ──────────┬──────────── Resource B
                       │
                   Event (対照表)
                       │
                   明細・属性
```

## エンティティの詳細分類

### Resource の下位分類

Resource はさらに以下のように分類される。

#### 1. 基本 Resource

独自のコード体系を持ち、業務上独立して管理されるもの。

```
  [R] 顧客
  ─────────
  顧客コード (個体指定子)
  顧客名
  住所
  電話番号
```

#### 2. 再帰的 Resource（自己参照）

同一 Resource 内での階層・関係を表すもの。

```
  [R] 組織
  ─────────
  組織コード (個体指定子)
  組織名
  上位組織コード → [R] 組織  （再帰参照）
```

#### 3. 対照 Resource（Resource 間の静的対応）

2つの Resource の間の**恒常的な対応関係**を表す。
Event と異なり、時間軸を持たない（いつ発生したかという概念がない）。

```
  [R] 顧客    [R] 担当者
      │            │
      └──┬─────────┘
         │
  [対照R] 顧客担当
  ────────────────
  顧客コード + 担当者コード (個体指定子)
  担当開始日
```

> **Event との区別**: 対照 Resource は「事実の記録」であり、
> Event は「出来事の記録」である。受注は起きるが、顧客担当は「起きる」のではなく「在る」。

### Event の下位分類

#### 1. 基本 Event（対照表）

2つ以上の Resource 間の出来事を記録する。
TM の最も中心的な概念。

```
  [R] 顧客 ──────────┬──────────── [R] 商品
                      │
                 [E] 受注
                 ──────────
                 受注番号 (個体指定子)
                 ├─ 顧客コード (→ R:顧客)
                 ├─ 受注日
                 └─ 合計金額
```

個体指定子のパターン：
- **参照複合型**: `顧客コード + 商品コード + 受注日`
- **代理キー型**: `受注番号`（連番やUUID）

実務上は代理キーを使うことが多いが、
TM の概念的には「どの Resource の組み合わせで識別されるか」が本質。

#### 2. 明細 Event（Event の内訳）

Event に対する明細行。Event × Resource の対応を表す。

```
  [E] 受注 ──────────┬──────────── [R] 商品
                      │
                 [E] 受注明細
                 ──────────────
                 受注番号 + 明細番号 (個体指定子)
                 ├─ 商品コード (→ R:商品)
                 ├─ 数量
                 └─ 金額
```

#### 3. 再帰的 Event

Event に対する後続 Event。業務フローの連鎖を表す。

```
  [E] 受注 ──────▶ [E] 出荷 ──────▶ [E] 請求

  受注と出荷は「同じ取引」の異なるフェーズ
  出荷の個体指定子は受注番号を含む（or 参照する）
```

## 導出規則

TM の強力な点は、要件記述から機械的にモデルを導出する規則が存在することである。

### 規則1: 名詞の抽出と分類

要件文から名詞を抽出し、Resource か Event かを判定する。

```
要件: 「顧客が商品を注文する。注文には複数の商品を含められる。」

名詞の抽出:
  ・顧客   → 独立して存在する → Resource
  ・商品   → 独立して存在する → Resource
  ・注文   → 顧客と商品の対応 → Event
```

**判定基準:**
- 「〜が〜する」の主語・目的語 → Resource の候補
- 「〜する」という動作・行為そのもの → Event の候補
- それ単独で存在できるか？ → Yes なら Resource、No なら Event

### 規則2: 個体指定子の決定

各エンティティに対し、個体を一意に特定する識別子を決定する。

```
[R] 顧客 → 顧客コード
[R] 商品 → 商品コード
[E] 注文 → 注文番号（= 顧客コード × 商品コード × 日時 の代理キー）
```

- Resource の個体指定子は**単独**
- Event の個体指定子は**参照先 Resource の識別子を含む**（概念的に）

### 規則3: 対照表（Event）の構成

Event は必ず 2つ以上の Resource を参照する。
この制約を明示することで、モデルの正規性を担保する。

```
[E] 注文
  参照: [R] 顧客 (誰が)
  参照: [R] 商品 (何を)  ← 明細経由の場合もある
  固有属性: 注文日, 合計金額
```

### 規則4: 明細の展開

「1つの Event に対して複数の〜がある」という要件は、明細 Event を導出する。

```
「1つの注文に複数の商品を含める」
  → [E] 注文明細 を導出
  → 個体指定子: 注文番号 + 明細番号
  → 参照: [R] 商品
  → 固有属性: 数量, 単価, 金額
```

### 規則5: 再帰的構造の識別

「〜の中に〜がある」「〜の上位に〜がある」→ 再帰的 Resource

```
「部門の中に下位部門がある」
  → [R] 部門 に 上位部門コード（自己参照）を追加
```

### 規則6: 業務フローの連鎖

時間的に連続する Event の連鎖を識別する。

```
受注 → 出荷 → 請求 → 入金

各 Event は前段の Event を参照する。
  [E] 出荷.受注番号 → [E] 受注
  [E] 請求.出荷番号 → [E] 出荷
```

## 正規化との関係

TM の規則に従ってモデリングすると、結果的に**第3正規形以上**のモデルが自然に得られる。

これは偶然ではなく、TM の分類規則が正規化の原理と同根であるため：

| TM の規則 | 対応する正規化原理 |
|---|---|
| 個体指定子の単独性（Resource） | 1NF: 原子的な値 |
| Event は Resource を参照する | 2NF: 部分関数従属の排除 |
| 属性は個体指定子に完全従属 | 3NF: 推移的関数従属の排除 |
| 対照 Resource の分離 | BCNF |

## 記法（ダイアグラム表現）

### エンティティの表記

```
┌─────────────────────┐
│ [R] 顧客            │  ← 種別 + エンティティ名
├─────────────────────┤
│ *顧客コード          │  ← * は個体指定子（PK）
├─────────────────────┤
│  顧客名              │  ← 属性
│  住所                │
│  電話番号            │
└─────────────────────┘

┌─────────────────────┐
│ [E] 受注             │
├─────────────────────┤
│ *受注番号            │
├─────────────────────┤
│  顧客コード (→顧客)  │  ← 参照（外部キー）
│  受注日              │
│  合計金額            │
└─────────────────────┘
```

### 関連線の表記

```
  [R] 顧客 ─────1:N──── [E] 受注 ────1:N──── [E] 受注明細 ────N:1──── [R] 商品

  R → E : 常に 1:N（1つの Resource が複数の Event に参加）
  E → 明細E : 常に 1:N（1つの Event が複数の明細を持つ）
  明細E → R : 常に N:1（各明細が1つの Resource を参照）
```

### 全体配置の慣例

```
  ┌────────────────────────────────────────────────┐
  │ Resource 層（上段）                              │
  │                                                │
  │  [R]顧客     [R]商品     [R]倉庫     [R]社員    │
  ├────────────────────────────────────────────────┤
  │ Event 層（下段）                                │
  │                                                │
  │  [E]受注 ─── [E]受注明細                        │
  │       │                                        │
  │  [E]出荷 ─── [E]出荷明細                        │
  │       │                                        │
  │  [E]請求 ─── [E]請求明細                        │
  └────────────────────────────────────────────────┘

  Resource は上、Event は下に配置するのが慣例。
  業務フローは上から下（時系列順）に流れる。
```

## VEA マトリクス（検証ツール）

TM では **VEA (Verify Entity and Attribute) マトリクス** を使って
モデルの完全性と正確性を検証する。

```
                    [R]顧客  [R]商品  [E]受注  [E]受注明細
  ─────────────────────────────────────────────────────
  顧客コード          PK       -       FK        -
  顧客名              ○       -        -        -
  商品コード           -       PK       -       FK
  商品名               -       ○        -        -
  単価                 -       ○        -        -
  受注番号             -        -       PK      FK(部分)
  受注日               -        -       ○        -
  明細番号             -        -        -       PK(部分)
  数量                 -        -        -       ○
  金額                 -        -        -       ○
```

検証ポイント：
- すべての属性がいずれかのエンティティに所属しているか
- 1つの属性が複数のエンティティに重複していないか（FK を除く）
- 個体指定子が適切に設定されているか

## 具体例：EC サイトの受発注

### 要件

```
・顧客が商品を注文できる
・1回の注文に複数の商品を含められる
・各商品には在庫がある（倉庫ごとに管理）
・注文が確定すると出荷指示が作られる
・出荷が完了すると請求が発行される
```

### Step 1: 名詞の抽出と分類

| 名詞 | 分類 | 理由 |
|------|------|------|
| 顧客 | Resource | 独立して存在、顧客コードで識別 |
| 商品 | Resource | 独立して存在、商品コードで識別 |
| 倉庫 | Resource | 独立して存在、倉庫コードで識別 |
| 注文 | Event | 顧客×商品の対応（出来事） |
| 在庫 | Event | 倉庫×商品の対応（状態） |
| 出荷 | Event | 注文の後続イベント |
| 請求 | Event | 出荷の後続イベント |

### Step 2: 対照表の構成と明細展開

```
  [R] 顧客          [R] 商品          [R] 倉庫
  ──────────        ──────────        ──────────
  *顧客コード        *商品コード        *倉庫コード
   顧客名            商品名            倉庫名
   住所              単価              所在地
   電話番号          カテゴリ

       │                │    │            │
       │                │    │            │
       ▼                ▼    ▼            ▼
  [E] 注文          [E] 注文明細     [E] 在庫
  ──────────        ──────────────  ──────────────
  *注文番号          *注文番号        *倉庫コード
   顧客コード(FK)    *明細番号         *商品コード
   注文日             商品コード(FK)    在庫数量
   合計金額           数量             最終更新日
                      金額
       │
       ▼
  [E] 出荷
  ──────────
  *出荷番号
   注文番号(FK)
   出荷日
   出荷先住所
       │
       ▼
  [E] 請求
  ──────────
  *請求番号
   出荷番号(FK)
   請求日
   請求金額
   支払期限
```

### Step 3: 検証

- すべての Resource は単独の個体指定子を持つ ✓
- すべての Event は Resource（または先行 Event）を参照する ✓
- 「1注文に複数商品」→ 注文明細で展開済み ✓
- 業務フロー: 注文 → 出荷 → 請求 の連鎖が表現されている ✓
- 在庫は倉庫×商品の対照表（時間軸を持たない → 対照 Resource 的性質も議論の余地あり）

## TM 手法の制約と限界

### 手法が強い領域

- **業務系システム**（受発注、在庫管理、会計など）：R/E の分類が自然に適合
- **正規化の担保**：規則に従えば自動的に高度な正規形に到達
- **要件の曖昧さの発見**：R/E に分類できないものは要件が不明確な証拠

### 手法が弱い領域

- **状態遷移が複雑なドメイン**：Event の「状態」をどう扱うか（ステートマシンの表現）
- **非構造データ**：JSON/ドキュメント型のデータは R/E 分類に馴染みにくい
- **多対多の高次関係**：3つ以上の Resource が絡む Event の表現が複雑になる

### 原典との距離について

本文書は佐藤正美氏の著作に基づく理解を整理したものであり、
原典の厳密な定義とは差異がある可能性がある。
特に以下の点は原典を参照されたい：

- 「みなし entity」の扱い
- 帰属 (aP) と参照 (rP) の厳密な区別
- HDR-DTL 構造の形式的定義
- NULL 制約に関する議論
