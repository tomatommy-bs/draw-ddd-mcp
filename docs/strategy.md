### 1. 業界内の競争 — ★★☆☆☆ 低い

「TM × AI × MCP」の交差点にはほぼ誰もいない。

| 領域 | 既存プレイヤー | draw-ddd-mcp との重なり |
|------|--------------|----------------------|
| ER モデリング | dbdiagram.io, DrawDB, MySQL Workbench | 図を描くだけ。構文チェックなし |
| DDD ツール | Context Mapper, Miro テンプレート | TM なし。手動。AI なし |
| TM モデリング | 佐藤正美氏の書籍・Excel | デジタルツール化されていない |
| AI DB 設計 | ChatGPT + 手動 | MCP 連携なし。構文論なし |

TM 手法自体がニッチであり、それをデジタルツール化した競合が存在しない。ブルーオーシャンだが、裏を返せば市場が立証されていない。

### 2. 新規参入の脅威 — ★★★☆☆ 中程度

技術的な参入障壁は低い。知識的な参入障壁は高い。

- **低い壁**: MCP サーバー + React GUI は汎用技術。DrawDB のフォークで誰でも作れる
- **高い壁**: TM 手法の理論を正しく実装するには深い理解が必要。`docs/tm/` に体系化した知識体系自体が参入障壁
- **AI ベンダーの動き**: Anthropic / OpenAI が DB 設計支援を内蔵する可能性はあるが、TM という特定手法に特化する動機がない

最大のリスク: 大手 AI ベンダーが「汎用 DB 設計 AI」を出したとき、TM の構文論的優位性を説明できないと埋もれる。

### 3. 代替品の脅威 — ★★★★☆ 高い

最大の脅威。「TM を使わない」という選択肢が常にある。

```
draw-ddd-mcp の代替手段:

  ┌─────────────────────────────────────────────────┐
  │  "AI に直接聞けばいい"                            │
  │  ChatGPT「受注管理の DB 設計して」→ DDL が出る    │
  │  構文論なし、検証なし、だが 30 秒で終わる          │
  └─────────────────────────────────────────────────┘
                    vs
  ┌─────────────────────────────────────────────────┐
  │  draw-ddd-mcp                                    │
  │  TM で構造化 → 構文チェック → ERD + DDD 導出      │
  │  正確だが、学習コスト + セットアップが必要          │
  └─────────────────────────────────────────────────┘
```

| 代替品 | 脅威度 | draw-ddd-mcp の対抗 |
|--------|--------|-------------------|
| ChatGPT に直接 DDL を書かせる | 高 | 規模が大きくなると破綻する。構文チェックがない |
| Figma / Miro で手描き | 中 | 図は描けるが検証できない |
| Excel で TM を手書き | 中 | TM の価値は認めているが DX が悪い |
| 既存 ER ツール + 経験則 | 中 | R/E の区別がなく、構造の妥当性検証ができない |

対抗の鍵: 「小規模なら AI 直接で十分だが、事業構造が複雑になるほど TM の構文チェックが効いてくる」というスケーリングの優位性を示すこと。

### 4. 売り手の交渉力 — ★★★★☆ 高い

AI プロバイダーへの依存が構造的に大きい。

```
依存関係:

  Claude API ──────▶ MCP プロトコル ──────▶ draw-ddd-mcp
  (Anthropic)         (Anthropic)

  Anthropic が価格改定 / MCP 仕様変更 → 直撃
```

- **Claude / MCP への一本足**: 現状 Anthropic のエコシステムに完全依存。MCP 仕様の変更、Claude の価格改定が直接影響する
- **緩和策**: MCP は OSS プロトコルであり、他の LLM プロバイダーも採用し始めている。Backend 層が MCP を抽象化しているので、LLM の差し替えは理論上可能
- **TM 知識の独立性**: `docs/tm/` の理論体系自体は LLM 非依存。これがコア資産

### 5. 買い手の交渉力 — ★★★☆☆ 中程度

- **スイッチングコスト**: TM モデルを構築した後の移行は難しい（TM 図を他ツールに持ち出す標準フォーマットがない）→ ロックイン効果あり
- **価格感度**: ローカルツール＋OSS なら無料。AI の API コストのみ。買い手の価格抵抗は低い
- **ただし**: ターゲットが「エンジニア + プロダクトメンバー」の場合、プロダクト側に TM を学ぶ意欲がなければ採用されない。買い手の「やる気」が最大の変数

## 注力戦略: AI 提案品質による差別化（Layer 3）

### なぜ Layer 3 か

知識のプロダクト埋め込みには3段階ある：

```
  Layer 1: docs に書いてある       ← 読めば真似できる（低障壁）
  Layer 2: バリデーションに実装     ← コードを読めば真似できる（中障壁）
  Layer 3: AI の提案品質に反映     ← 模倣困難（高障壁）
```

Layer 3 は「TM 理論の体系化 × MCP ツール設計 × prompt の質」の三位一体であり、どれか一つを真似しても再現できない。ツールとしての完成度より先に、「AI + TM でモデリングすると、こんなに正確なモデルが出る」という体験を作ることが最も ROI が高い。

### Layer 3 の構成要素

### 具体的な方針

| 要素 | 現状 | 強化方針 |
|------|------|---------|
| TM 知識体系 | docs/tm/ に体系化済み | AI が参照しやすい形に維持・拡充する |
| MCP ツール設計 | 21 ツール実装済み | ツールの description に TM 理論の根拠を埋め込み、AI が「なぜそう構造化すべきか」を説明できるようにする |
| AI の振る舞い | 未定義 | 期待する提案パターンを定義する（下記） |

### AI に期待する提案パターン

```
ユーザー: 「EC サイトの受注管理を作りたい」

期待する AI の振る舞い:

  1. 事業概念の抽出
     「登場するモノ(R)とコト(E)を整理します」
     → 顧客(R), 商品(R), 受注(E), 出荷(E), 請求(E)

  2. TM 構文に基づくモデル構築
     → E→R 参照を自動設定（受注→顧客, 受注→商品）
     → E-E 関係の判定（受注:出荷 = 1:N → 直接参照）
     → 必要に応じて C/M を提案

  3. 構文チェックの実行と説明
     「受注(E)が商品(R)を直接参照していますが、
      受注明細(E)を経由すべきではないですか？
      1回の受注で複数商品を扱う場合、受注:商品 = N:M となり
      Mapping が必要です」

  4. フィードバックに基づく修正
     ユーザーの FB を受けて修正し、再度構文チェック
```

このパターンの品質が高ければ高いほど、「AI に直接聞くより draw-ddd-mcp を使った方が正確」という体験が生まれ、代替品の脅威が下がる。
