# Event (E)

根拠: 公理 A1, A5, A6

## 定義

ある時点で発生する「こと」。時間に依存し、発生するたびにレコードが増える。

## 判定基準

- 「それはいつ起きたか？」→ 問えるなら Event
- 「時間とともにレコードが増えるか？」→ Yes なら Event
- 「それは Resource の状態を変えるか？」→ Yes なら Event の候補

## 他エンティティとの関係

Event は必ず1つ以上のエンティティ（R または E）を参照する。単独では存在しない。

参照先は Resource に限らず、Event でもよい。ただし、参照の連鎖を辿ると最終的に少なくとも1つの Resource に到達しなければならない（Event だけで閉じたループは無意味）。

```
受注 (E) → 顧客 (R)     「誰が」注文したか（E → R 直接参照）
受注 (E) → 品目 (R)     「何を」注文したか（E → R 直接参照）
請求 (E) → 受注 (E)     「どの注文に対して」請求したか（E → E 参照）
出荷 (E) → 受注 (E)     「どの注文に対して」出荷したか（E → E 参照）
```

請求や出荷のように Resource を直接参照しない Event は、受注を経由して間接的に顧客・品目(R)に到達する。

### E-E の関係と cardinality

- **1:N**: E2 が E1 を直接参照する（E2 の左列に E1 の own が (R) で出現）
- **N:1 / N:M**: Mapping (M) を間に定義する

## 時間属性

Event は「いつ起きたか」を示す時間属性（日付・日時）を持つべきである。

- 時間属性の欠如は warning（error ではない）
- 典型的な時間属性名: `受注日`, `出荷日時`, `支払日`, `登録日時`
- 属性分類上は「状態記述子」に該当する

## subtype 別の構造

### basic

最も基本的な Event。1つ以上の R（または E）を参照し、固有指定子を持つ。

```
受注 (E, basic)
┌──────────────────┬────────────┐
│  受注              │     E      │
├──────────┬───────┴────────────┤
│受注NO    │ 受注日              │
│顧客NO(R) │ 受注数              │
│品目コード(R)│                   │
└──────────┴────────────────────┘
  左列: own + reference
```

### subset

親 Event の特化。

```
クレジット支払 (E, subset)
┌──────────────────┬────────────┐
│  クレジット支払     │     E      │
├──────────┬───────┴────────────┤
│支払NO(R) │ カード番号          │
│          │ 承認番号             │
└──────────┴────────────────────┘
  親: 支払 (E, basic)
```
