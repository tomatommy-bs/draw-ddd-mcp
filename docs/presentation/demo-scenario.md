# デモシナリオ: draw-datamodel

> 所要時間: 約 5〜7 分
> 対象スライド: 「デモ: AI とモデリングする」

---

## 前提

- `pnpm dev` で backend + GUI が起動済み
- Claude Code から MCP ツールが接続済み
- ダイアグラムは空の状態からスタート

---

## プロンプト

```
メルカリのような C2C フリマアプリの取引フローをモデリングしてほしい。
ユーザーが商品を出品して、別のユーザーがそれを購入し、出品者が発送する。
```

---

## シナリオ

### Act 1: モデルの構築（AI が提案 → GUI に即反映）

**話しながらやること**: 自然言語の一言から、AI がドメインの概念を「モノ」と「コト」に分類してモデルを作る様子を見せる。

#### 1-1. リソース（モノ）を追加

| エンティティ | 色 | 属性 | 個体指定子 |
|---|---|---|---|
| ユーザー | 青 `#2563eb` | ユーザーID, ニックネーム, 評価スコア | ユーザーID |
| 商品カテゴリ | 青 `#2563eb` | カテゴリID, カテゴリ名 | カテゴリID |
| 配送エリア | 青 `#2563eb` | エリアID, エリア名 | エリアID |

→ **オートレイアウト実行**

**見せ場**: GUI に青いノードが 3 つ出現し、整列される瞬間。
「"ユーザー", "商品カテゴリ", "配送エリア" は時間に依存しない "モノ" なので、リソースとして青で配置されます。」

#### 1-2. イベント（コト）を追加

| エンティティ | 色 | 属性 | 個体指定子 |
|---|---|---|---|
| 出品 | 赤 `#dc2626` | 出品ID, 商品名, 価格, 出品日時 | 出品ID |
| 購入 | 赤 `#dc2626` | 購入ID, 購入日時, 支払方法 | 購入ID |
| 発送 | 赤 `#dc2626` | 発送ID, 発送日, 追跡番号, 配送方法 | 発送ID |

→ **オートレイアウト実行**

**見せ場**: 赤いノードが下段に出現し、リソースと視覚的に区別される。
「"出品", "購入", "発送" は特定の時点で起きる "コト" なので、イベントとして赤で配置されます。」

#### 1-3. 参照（関連）を追加

| 参照元 | → | 参照先 | カーディナリティ | ラベル |
|---|---|---|---|---|
| 出品 (E) | → | ユーザー (R) | N:1 | 出品者 |
| 出品 (E) | → | 商品カテゴリ (R) | N:1 | カテゴリ |
| 購入 (E) | → | ユーザー (R) | N:1 | 購入者 |
| 購入 (E) | → | 出品 (E) | N:1 | 購入対象 |
| 発送 (E) | → | 出品 (E) | N:1 | 発送対象 |
| ユーザー (R) | → | 配送エリア (R) | N:1 | 居住エリア |

→ **オートレイアウト実行**

**注目ポイント（2 つ）**:

1. AI は TM の構文ルールを理解し、参照を追加する際に **個体指定子を自動で伝播** している。出品にはユーザーID・カテゴリID が、購入・発送には出品ID が自動的に付与される。

2. しかし最後の **ユーザー→配送エリア** で、ユーザーノードに **赤枠** が表示される。

---

### Act 2: 構文チェック（TM の文法ルールが誤りを検出）

**話しながらやること**: AI が自動で多くのルールに従えることと、それでも検出される構造的な誤りがあることを対比して見せる。

#### 2-1. 自動伝播の確認

出品ノードをクリックして属性を見せる。

**解説**:
「出品の属性を見てください。出品ID だけでなく、ユーザーID とカテゴリID が自動で追加されています。TM の文法では、イベントがリソースを参照するとき、"誰の" "何の" を特定するための個体指定子を持たなければなりません。AI はこのルールを理解し、自動的に正しく伝播しています。」

#### 2-2. 違反の確認

**TM-G-002（R→R 直接参照の禁止）**:
- **ユーザー** ノードが赤枠: ユーザー→配送エリア は「モノ」同士の直接参照

**解説**:
「一方で、ユーザーと配送エリアの関係には赤枠が出ています。TM では "モノ" 同士を直接参照できません。"ユーザーがどのエリアに住んでいるか" は一見正しそうですが、引っ越しや住所変更など、時点によって変わりうる関係です。こうした関係はイベントや対照表を経由して表現すべきです。

コードの型チェッカーと同じように、人間が見落としがちな構造の誤りをツールが自動で指摘してくれます。」

---

### Act 3: 修正（AI がルールに従って修正 → 赤枠が消える）

#### 3-1. R→R 直接参照を削除

ユーザー→配送エリア の参照を削除する。

**見せ場**: 参照を削除した瞬間にユーザーノードの赤枠が消える。

**解説**:
「"ユーザーはどのエリアに住んでいるか" という関係は、住所登録というイベントや、ユーザーとエリアをつなぐ対照表を通じて表現します。直接紐づけると引っ越しの履歴が残らない設計になってしまう。TM の構文ルールが、こうした設計上の落とし穴を未然に防いでくれます。」

#### 3-2. バリデーション実行

```
tm_validate_model を実行
```

**結果**: エラーなし。バリデーション成功。

---

### Act 4: ユースケース分析（属性の所在を可視化）

**話しながらやること**: 構造が正しいことを確認した次のステップとして、「この属性は本当にこのエンティティにあるべきか？」をユースケース単位で検証できることを見せる。

#### 4-1. ユースケース「購入する」を登録

対象:
- **ユーザー**: 全体（購入者の情報が必要）
- **出品**: 出品ID, 商品名, 価格のみ（出品日時やカテゴリは購入時に不要）
- **購入**: 全体

**見せ場**: 
- ユーザー・購入は黄色枠でハイライト
- 出品は指定した 3 属性だけ黄色背景
- 商品カテゴリ・発送は薄く dimmed

**解説**:
「ユースケース "購入する" に関係するエンティティと属性だけをハイライトしました。出品の "出品日時" や "カテゴリ" は購入フローでは使わないので、ハイライトされていません。もしここに購入フローでしか使わない属性が出品に含まれていたら、それは出品ではなく購入に置くべきかもしれない。こうした分析が GUI 上でできます。」

#### 4-2. ユースケース「発送する」に切り替え

対象:
- **出品**: 出品ID のみ
- **発送**: 全体

**見せ場**: ハイライトが即座に切り替わる。購入やユーザーが dimmed になり、出品は出品ID だけがハイライト。

**解説**:
「今度は "発送する" に切り替えました。発送フローでは出品の詳細は不要で、出品ID だけで紐づいている。このように、ユースケースを切り替えながら "どの属性がどの場面で使われるか" を可視化し、属性の配置が適切かどうかを検証できます。」

#### 4-3. GUI パネルからの切り替えデモ

Toolbar の Use Cases ボタンを押して右パネルを開く。

**見せ場**: 保存されたユースケース一覧が表示され、クリックでハイライトが切り替わる。再クリックで解除。

**解説**:
「登録したユースケースは GUI のパネルに一覧化されます。PdM とエンジニアが同じ画面を見ながら、"この機能ではどのデータを使うか" を指差しで確認できます。」

---

### Act 5: ユビキタス言語（用語集の自動蓄積）

**話しながらやること**: モデリング中に生まれた用語が Glossary に蓄積されることを見せる。

#### 5-1. 用語を登録

| 用語 | 定義 | 却下された別名 | 却下理由 |
|---|---|---|---|
| ユーザー | フリマアプリの利用者。出品者にも購入者にもなる | 会員 | 未登録の閲覧者と区別しにくいため |
| | | 顧客 | C2C モデルでは売り手でもあるため |
| 出品 | ユーザーが商品を販売するために登録する行為 | 登録 | 「何を」登録するか曖昧なため |
| | | リスティング | 日本語のユビキタス言語として「出品」を採用 |
| 購入 | ユーザーが出品された商品を買う行為 | 注文 | BtoC 的な表現。C2C では「購入」が自然 |
| 発送 | 出品者が購入された商品を配送に出す行為 | 出荷 | BtoC/物流の用語。個人間取引では「発送」が適切 |

**見せ場**: 用語がエンティティと双方向リンクされる。却下理由が記録される。

**解説**:
「なぜ "注文" ではなく "購入" なのか。メルカリは C2C なので、BtoC 的な "注文" ではなく "購入" が自然です。こうした判断の理由まで記録されます。DDD が説くユビキタス言語を、データモデルという構造の上に構築する。これがエンジニアと PdM の間の "言葉のブレ" を防ぎます。」
