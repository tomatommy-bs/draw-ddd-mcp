# Vision

## 一言で

TM（T字型ER手法）の構文論と AI の意味理解を組み合わせ、事業構造の分析から DB 設計・DDD 集約定義までを一貫して支援するツール。

## 何が嬉しいのか — Figma や汎用 Draw ツールとの違い

汎用の作図ツールは「自由に描ける」が、描いた図が正しいかどうかは人間が判断するしかない。draw-ddd-mcp は TM 図・ERD・Aggregate Design Canvas に特化することで、以下を提供する：

- **構文チェック**: TM の公理に従って、事業分析の構造的な誤りをリアルタイムに検出する
- **自動導出**: 少数の意味論的判断（R/E の分類、関係の有無）から、FK の方向・中間テーブル・参照指定子を自動生成する
- **一気通貫**: 1つのモデルから TM 図 → ERD → DDD 集約定義 → DDL まで導出する

汎用ツールでは「自由に描けるが誤りに気づけない」。特化ツールでは「制約の中で描くからこそ誤りに気づける」。

- **利用形態**: 1台の PC を会話の媒体として共有し、AI を介して対話する
- **初期スコープ**: ローカル環境でのみ動作する小規模ツール

## 3つの柱

### 1. AI を介したエンジニアとプロダクトメンバーの意思疎通

事業構造を理解するための手法として TM を採用し、AI を用いてエンジニアとプロダクトメンバーの意思疎通とシステムの一貫性を担保する。

```
プロダクトメンバー                        エンジニア
「顧客が注文して                         CREATE TABLE orders (
 配送される」                             customer_id INT REFERENCES...
     │                                        ▲
     ▼                                        │
┌─────────────────────────────────────────────────┐
│                    AI + TM                       │
│                                                  │
│  自然言語  ──▶  TM モデル  ──▶  DDL / コード     │
│  (意味論)       (構文論)        (実装)            │
└─────────────────────────────────────────────────┘
```

TM モデルが共通言語として機能することで、「プロダクトが語る事業の言葉」と「エンジニアが書くコードの構造」の間にある翻訳ロスを最小化する。

### 2. DB 設計の正確性

ソフトウェア開発の基盤となる DB 設計を、構文規則に基づいて正確に行う。

| TM の構文ルール | DB 設計への効果 |
|---|---|
| R は他を参照しない | マスタテーブルの独立性保証 |
| E → R は必ず E 側が参照 | FK の方向が自動決定 |
| R-R は直接参照不可、C を経由 | 多対多の中間テーブルが構造的に必然 |
| 構造決定性（A7） | 参照指定子が自動導出 → FK の手動設計ミスを排除 |
| サブセット（A8） | 継承関係の正規化が構文で決まる |

人間が下す判断（R か E かの分類、エンティティ間の関係の有無）を最小化し、残りの構造を TM の文法規則から自動導出する。

### 3. 構文論 × 意味論の二重分析

事業分析を意味論だけで考えるのではなく、TM という構文論も用いて分析する。

```
┌──────────────────────────────────────────────┐
│  意味論的判断（人間 + AI が協働）              │
│  ・これは R（モノ）か E（コト）か？            │
│  ・このエンティティ間に関係はあるか？           │
│  ・サブタイプに分解すべきか？                  │
├──────────────────────────────────────────────┤
│  構文論的帰結（TM ルール + ツールが自動処理）   │
│  ・参照の方向                                │
│  ・参照指定子の生成                           │
│  ・中間テーブル（C / M）の必要性               │
│  ・モデルのバリデーション                      │
└──────────────────────────────────────────────┘
```

従来のモデリング（ER 図、UML）は意味論ドリブン — 人間がすべての構造を判断する。TM の革新は構文論の導入にあり、少数の意味論的判断から残りを構文的に導出できる。

### TM の可視性

TM 図はプロダクトメンバーにもそのまま見せる。TM は ERD よりも非エンジニアフレンドリーである：

- R（モノ）/ E（コト）の区別は日常語に対応する — 「顧客」はモノ、「注文」はコトと直感的に理解できる
- T字型の左列（誰/何）/ 右列（属性）の構造は、表計算に慣れた人なら読める
- AI が自然言語で補足説明することもできるが、TM 図自体を隠す必要はない

### AI の役割

生成 AI の利点を活かし、AI は積極的にモデルを提案する：

- 「受注管理を作りたい」→ AI が R/E/C/M の初期モデルを提案
- 人間が TM 図を見ながらフィードバック（「この関係は違う」「これは別のエンティティだ」）
- AI が修正、または人間が GUI で直接編集
- TM の構文チェックが常に働き、構造的な誤りはツールが指摘する

### TM → DDD の導出

現時点の方針：

- **構文論での導出**: 探索中。TM の構造規則から集約境界を自動推論できる可能性があるが、まだ確立されていない
- **当面のアプローチ**: AI が TM モデルを読み、意味論的に DDD 集約定義を提案する。人間とのやりとりで洗練する
- **将来の可能性**: 構文論的な導出ルールが見つかれば、Phase 2a（DB 設計）と同様に自動化の余地がある
