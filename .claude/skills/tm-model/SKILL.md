# TM モデリングスキル

このスキルは draw-ddd-mcp の MCP ツール群を使い、TM (T字型ER手法) によるドメインモデリングを行います。

## 起動時の処理

1. **GUI 接続確認**: `tm_get_diagram` を呼んで GUI が接続されているか確認する。失敗した場合は `pnpm dev` でサーバーと GUI を起動するよう案内する。

2. **現状把握**: ダイアグラムにエンティティが既にある場合はその内容を要約する。空なら新規モデリングとして進める。

3. **ユーザーの意図を確認**: ユーザーが `/tm-model` に引数を渡した場合（例: `/tm-model 受注管理`）はそのドメインでモデリングを開始する。引数がない場合はどのドメインをモデリングするか聞く。

## モデリングの進め方

### tm-modeler エージェントを活用する

モデリングの実作業は `tm-modeler` エージェントに委任できます。特に：

- **ユーザーへの質問（`tm_prompt_user`）はブロッキング呼び出し**なので、background sub-agent に委任することでメイン処理を止めずに済む
- 複数エンティティの並行作成なども sub-agent で効率化できる

### モデリングフロー

**Phase 1: ドメイン分析**
- ユーザーの要件からドメインの主要概念を抽出
- Resource（モノ）と Event（コト）を識別
- 不明点があれば `tm_prompt_user` で GUI 経由で確認

**Phase 2: モデル構築**
- Resource を先に作成（上段配置）
- Event を作成し Resource への参照を設定（下段配置）
- 各エンティティの属性と個体指定子を設定
- 必要に応じて Correspondence / Recursive / Mapping を追加

**Phase 3: 検証と整理**
- `tm_validate_model` でモデルの整合性チェック
- `tm_auto_layout` でレイアウト整理
- 検証エラーがあればユーザーと相談して修正

### ユーザーへの質問パターン

GUI 経由の質問（`tm_prompt_user`）は以下の場面で使う：

```
# ドメイン概念の判断
tm_prompt_user({
  title: "「商品カテゴリ」の扱い",
  message: "商品カテゴリは独立した Resource として管理しますか？それとも商品の属性として扱いますか？",
  type: "select",
  options: ["独立した Resource", "商品の属性"]
})

# 追加情報の収集
tm_prompt_user({
  title: "受注の属性",
  message: "受注イベントに含めたい属性を教えてください（例: 受注日, 数量, 金額）",
  type: "text"
})

# 操作の確認
tm_prompt_user({
  title: "モデルの初期化",
  message: "既存のダイアグラムを消去して新規にモデリングを始めますか？",
  type: "confirm"
})
```

## TM 手法の基本ルール

- **Resource** は他を参照しない。参照されるだけ。
- **Event** は必ず1つ以上の Resource または Event を参照する。
- 参照元には参照先の個体指定子が自動コピーされる。
- 詳細は `docs/tm/` の理論ドキュメントを参照。

## 注意事項

- モデリングの判断に迷うときは必ず `docs/tm/theory/` を読んで理論に立ち返ること
- ユーザーの業務知識を尊重し、一方的にモデルを決めないこと
- 大規模な変更の前には `tm_export_diagram` でバックアップを取ること
